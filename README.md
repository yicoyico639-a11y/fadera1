为设计一个深耕**去中心化AI训练协作**的项目：**如何让互不信任的参与方共同训练一个AI模型**。

---

### 项目名称：**Federa — 去中心化联邦学习激励协议**

**一句话简介**
一个基于零知识证明的联邦学习任务市场，让数据提供者和训练者在不可信环境中协作训练AI模型，并按贡献自动获得代币奖励。

---

### 要解决的问题

联邦学习允许多方在不共享原始数据的前提下协作训练模型，但其落地受困于三个"协作陷阱"：

- **搭便车与造假**：参与者可能提交随机梯度，假装参与训练却坐享奖励。
- **贡献度量不公**：无法精确衡量每个参与方提供的数据或算力对最终模型性能的真实边际贡献。
- **中心化协调者需信任**：传统联邦学习依赖的中央服务器，可能窃取模型或暗中偏袒特定节点。

Federa将联邦学习改造为一个**利益绑定、自动审计、贡献可验证的开放市场**：任务发起者质押奖金，训练节点质押参与，通过链上梯度验证和贡献评分，实现无需信任的全球AI协作训练。

---

### 核心创新点

| 创新维度 | 具体内容 |
|:---|:---|
| **可验证训练证明（PoT）** | 训练节点在本地执行梯度计算，并用RISC Zero zkVM生成ZK证明："本轮梯度更新是基于指定数据分区和当前全局模型正确计算得出的"，杜绝虚假梯度。 |
| **沙普利值贡献度分账** | 合约记录每轮验证通过的梯度更新，通过轻量级近似沙普利值算法，精确量化每个节点对模型性能提升的边际贡献，据此分配赏金。 |
| **隐私保护数据注册** | 数据提供者可将数据哈希与访问策略上链，训练节点通过TEE或密码学许可获取加密数据，确保原始数据不出域。 |
| **动态聚合与仲裁** | 聚合节点（任何人都可竞选，通过质押成为聚合者）收集梯度更新并合并为全局模型，若聚合者作恶，其质押将被罚没。链上仅记录模型性能的哈希演替，无需暴露全量参数。 |
| **模型进化NFT** | 全局模型的每个版本被铸造成一个状态NFT，记录贡献者列表、性能指标和版本血缘，成为可追溯的AI资产。 |

### 具体运作流程（演示场景）

1. **任务发布与质押**：任务发起者在Federa合约中开户，上传初始模型框架哈希，质押1000 USDC作为训练奖励池，并设定每轮验证奖励。
2. **数据提供者注册**：各自将加密的数据索引哈希、数据分布摘要上链，设定授权策略（如只允许在TEE中解密）。
3. **训练节点参与**：两名训练者Alice和Bob分别质押200 USDC，并获取授权访问的数据分区。在TEE或本地可信环境中，下载当前全局模型，执行一轮梯度下降，生成梯度更新向量。随后在RISC Zero zkVM内运行证明生成代码，证明"该梯度更新是对指定数据集在当前模型参数下计算损失并反向传播的正确结果"。
4. **提交与验证**：Alice和Bob分别将ZK证明和梯度更新的哈希提交至合约。合约验证证明后，将该轮梯度标记为有效，并记录提交节点地址。
5. **聚合与贡献分账**：聚合节点Charlie（通过质押竞选）收集到有效梯度后，执行联邦平均算法，生成新全局模型，并将新模型性能（如准确率）的哈希以及与上一版本的差异报告上链。合约根据沙普利值近似算法（比如比较"有Alice参与聚合"与"无Alice参与聚合"的边际准确率提升），计算出Alice的贡献度，按比例从奖池中释放赏金。
6. **罚没与挑战**：若有人恶意提交无效梯度，ZK证明会直接验证失败，该训练者被扣除部分质押金并重新分配给诚实节点。若聚合节点迟迟不聚合或提交错误聚合结果，任何节点均可发起挑战，通过提交独立聚合结果来仲裁，作恶者质押被罚没。

## 二、防作恶机制设计

### 2.1 作恶场景分类与威胁模型

| 角色 | 作恶类型 | 威胁等级 | 简述 |
|:---|:---|:---|:---|
| 训练节点 | 搭便车 | 高 | 提交零向量或极小梯度，几乎不贡献算力但分享奖励 |
| 训练节点 | 梯度造假 | 高 | 提交精心构造的恶意梯度，意图偏移模型方向（如后门攻击） |
| 训练节点 | 数据投毒 | 中 | 使用低质/错误数据训练，产生看似合法但有害的梯度 |
| 训练节点 | 梯度复制 | 中 | 多账号提交相同或线性相关梯度，放大自身份额 |
| 聚合节点 | 选择性聚合 | 高 | 故意忽略某些有效梯度，偏袒特定训练节点 |
| 聚合节点 | 聚合造假 | 高 | 提交的聚合结果并非联邦平均的正确计算 |
| 聚合节点 | 拖延不聚合 | 中 | 恶意拖延聚合时间，导致训练停滞 |
| 任务发起者 | 拒绝释放赏金 | 中 | 任务完成后不确认或恶意质疑有效训练结果 |
| 数据提供者 | 数据造假 | 中 | 注册的数据质量与描述不符，或提供随机噪声数据 |

### 2.2 多层防御体系

```
┌──────────────────────────────────────────────────────┐
│              Layer 1: ZK证明层（数学保证）            │
│  证明梯度更新的计算过程正确性，从数学上杜绝伪造梯度  │
├──────────────────────────────────────────────────────┤
│              Layer 2: 质押经济层（利益绑定）          │
│  所有参与者必须质押，作恶即罚没，提高攻击成本        │
├──────────────────────────────────────────────────────┤
│              Layer 3: 统计检测层（异常发现）          │
│  梯度范数检测、余弦相似度检测、分布偏离检测          │
├──────────────────────────────────────────────────────┤
│              Layer 4: 仲裁挑战层（社会共识）          │
│  任何人可发起挑战，通过独立重算验证，罚没作恶者      │
├──────────────────────────────────────────────────────┤
│              Layer 5: 信誉系统层（历史约束）          │
│  长期作恶者信誉归零，逐步被排除出系统                │
└──────────────────────────────────────────────────────┘
```

### 2.3 各层详细设计

#### Layer 1：ZK证明层

训练节点提交梯度时，必须同时提交RISC Zero zkVM生成的证明。证明的公开输入包括：

- `model_hash`：当前全局模型的哈希（训练开始时的快照）
- `data_commitment`：所使用数据分区的Merkle根承诺
- `gradient_hash`：生成的梯度更新向量的哈希
- `hyperparams_hash`：超参数（学习率、batch size等）的哈希

证明的私有输入包括：

- 实际的模型参数
- 实际的数据批次
- 实际的梯度向量
- 实际的超参数值

ZK电路验证的核心逻辑：

```
fn verify_training_step(
    model: Vec<f32>,       // 私有：当前模型参数
    data_batch: Vec<f32>,  // 私有：训练数据批次
    gradient: Vec<f32>,    // 私有：梯度更新
    lr: f32,               // 私有：学习率
) -> bool {
    // 1. 哈希模型参数，匹配公开的 model_hash
    assert!(hash(model) == public_model_hash);
    // 2. 哈希数据批次，匹配公开的 data_commitment
    assert!(hash(data_batch) == public_data_commitment);
    // 3. 前向传播计算损失
    let loss = forward(model, data_batch);
    // 4. 反向传播计算梯度
    let computed_gradient = backward(model, data_batch, loss);
    // 5. 验证提交的梯度与计算梯度一致（允许微小数值误差 ε）
    assert!(l2_norm(gradient - computed_gradient) < ε);
    // 6. 哈希梯度，匹配公开的 gradient_hash
    assert!(hash(gradient) == public_gradient_hash);
    return true;
}
```

**关键安全属性**：ZK证明保证了"正确执行"（correct execution），而非"结果有用"（useful result）。即证明可以验证梯度是按指定数据和模型正确计算出来的，但无法保证该梯度对模型性能有正面贡献——这需要Layer 3和贡献评分机制来补充。

#### Layer 2：质押经济层

**质押矩阵**：

| 角色 | 最低质押额 | 罚没条件 | 罚没比例 |
|:---|:---|:---|:---|
| 训练节点 | 200 USDC等值 | ZK证明验证失败 | 100% 质押 |
| 训练节点 | 200 USDC等值 | 梯度范数异常（≈0或极大） | 50% 质押 |
| 训练节点 | 200 USDC等值 | 多账号梯度复制被检出 | 100% 质押 |
| 聚合节点 | 500 USDC等值 | 聚合结果被挑战且验证为错 | 100% 质押 |
| 聚合节点 | 500 USDC等值 | 超时未提交聚合结果 | 30% 质押 |
| 聚合节点 | 500 USDC等值 | 选择性忽略有效梯度 | 100% 质押 |
| 任务发起者 | 赏金池全额 | 恶意拒绝确认有效结果 | 丧失争议赏金 |

**质押释放规则**：

- 训练节点：每轮训练结束后，验证通过的节点可提取质押+奖励；验证失败则罚没。
- 聚合节点：提交聚合结果后进入7天挑战期，无挑战则释放质押。
- 任务发起者：任务结束后剩余质押可提取。

#### Layer 3：统计检测层

ZK证明只能保证计算过程的正确性，无法判断梯度质量。统计检测层作为补充，通过链下计算+链上验证来识别低质或恶意梯度。

**检测项目**：

| 检测项 | 目标作恶 | 方法 | 阈值 |
|:---|:---|:---|:---|
| 梯度L2范数检测 | 搭便车（零/极小梯度） | 计算 \|\|g_i\|\|_2 | \|\|g_i\|\| < θ_min 则标记异常 |
| 梯度L2范数上限 | 梯度爆炸/恶意大梯度 | 计算 \|\|g_i\|\|_2 | \|\|g_i\|\| > θ_max 则标记异常 |
| 余弦相似度检测 | 梯度复制（多账号） | 计算 cos(g_i, g_j) 对所有i≠j | cos > θ_cos 则标记为疑似复制 |
| 梯度与全局方向一致性 | 对抗性梯度（后门） | 计算 cos(g_i, g_aggregate) | cos < θ_adv 则标记为疑似对抗 |
| 梯度方差检测 | 共谋攻击 | 计算所有梯度的 pairwise 距离方差 | 方差 < θ_var 则标记疑似共谋 |

**MVP简化方案**：黑客松阶段仅实现L2范数检测和余弦相似度检测，其余留作后续迭代。

**检测执行位置**：由聚合节点在链下执行统计检测，将检测结果和零知识证明一同提交至合约。合约验证统计检测证明后，排除异常梯度不参与聚合。

#### Layer 4：仲裁挑战层

当聚合节点或其他参与者被怀疑作恶时，任何人可以发起挑战。

**挑战流程**：

```
挑战者                     合约                       被挑战者
  │                         │                           │
  ├─ 提交挑战 + 质押 ──────>│                           │
  │                         ├─ 冻结被挑战者质押         │
  │                         ├─ 开启仲裁窗口(T=48h)      │
  │                         │                           │
  │<──── 仲裁开始事件 ──────┤──── 仲裁开始事件 ────────>│
  │                         │                           │
  │  链下独立重算聚合结果   │     链下独立重算聚合结果  │
  │                         │                           │
  ├─ 提交独立聚合证明 ────>│                           │
  │                         │<── 提交独立聚合证明 ──────┤
  │                         │                           │
  │                         ├─ 比较两份证明结果         │
  │                         ├─ 裁定胜负                 │
  │                         ├─ 罚没败方质押             │
  │<──── 裁定结果 ──────────┤──── 裁定结果 ───────────>│
```

**仲裁规则**：

- 挑战者需质押与被挑战者等额的资金作为挑战保证金。
- 仲裁期48小时，双方各提交一份聚合结果的ZK证明。
- 合约比较两份证明：若挑战者证明正确且被挑战者结果不同，被挑战者质押全部罚没，50%归挑战者，50%归奖池。
- 若挑战者证明错误或超时未提交，挑战者质押罚没，归被挑战者。
- 对于复杂争议（如数据质量争议），引入预选仲裁员委员会（7人随机选取，多数票裁决）。

#### Layer 5：信誉系统

每个地址维护一个链上信誉分数，初始为100分。

| 事件 | 信誉变化 |
|:---|:---|
| 成功完成一轮训练且贡献为正 | +2 |
| 梯度被统计检测标记异常 | -10 |
| ZK证明验证失败 | -50 |
| 被仲裁判定作恶 | -100（归零） |
| 成功发起有效挑战 | +5 |
| 挑战失败 | -20 |

**信誉影响**：

- 信誉低于30：无法参与高奖励任务
- 信誉低于10：无法竞选聚合节点
- 信誉为0：地址被临时封禁，需等待30天冷却后重新注册（信誉重置为50）

## 三、训练有效性验证体系

### 3.1 验证层次总览

训练有效性的验证不是单点判断，而是贯穿训练全生命周期的多层验证：

```
┌─────────────────────────────────────────────────────┐
│  有效性验证 = 计算正确性 + 梯度质量 + 模型提升      │
│                                                     │
│  计算正确性 ← ZK证明（Layer 1）                     │
│  梯度质量   ← 统计检测（Layer 3）                   │
│  模型提升   ← 贡献评分（Shapley值）                 │
└─────────────────────────────────────────────────────┘
```

### 3.2 ZK证明验证（计算正确性）

确保梯度是"诚实计算"的结果，而非凭空捏造。

**证明生成流程**：

1. 训练节点下载全局模型快照（通过IPFS哈希定位）
2. 获取授权数据分区的访问密钥
3. 在本地执行一轮或多轮梯度下降
4. 将完整的训练执行轨迹输入RISC Zero zkVM
5. zkVM生成STARK证明，证明执行轨迹满足训练电路约束
6. 将证明递归压缩为可在链上高效验证的形式

**证明生成耗时估计（MVP）**：

| 模型规模 | 数据量 | 证明生成时间 | 证明大小 | 链上验证Gas |
|:---|:---|:---|:---|:---|
| 小型（MLP, <1M params） | 1K samples | ~2 min | ~200KB | ~300K gas |
| 中型（CNN, ~10M params） | 10K samples | ~15 min | ~500KB | ~500K gas |

> **MVP策略**：黑客松阶段使用小型模型（如MNIST分类MLP），确保证明生成在可接受时间内完成。

### 3.3 统计质量检测（梯度质量）

验证梯度不仅"算对了"，而且"有营养"。

**具体检测逻辑**：

```python
def quality_check(gradients: dict[str, Vec], config) -> dict[str, str]:
    results = {}
    valid_grads = {}

    # 1. L2范数过滤
    for node_id, grad in gradients.items():
        norm = l2_norm(grad)
        if norm < config.norm_min:
            results[node_id] = "REJECTED: gradient_norm_too_small"
        elif norm > config.norm_max:
            results[node_id] = "REJECTED: gradient_norm_too_large"
        else:
            results[node_id] = "PASSED: norm_check"
            valid_grads[node_id] = grad

    # 2. 余弦相似度检测（梯度复制）
    ids = list(valid_grads.keys())
    duplicate_groups = []
    for i in range(len(ids)):
        for j in range(i+1, len(ids)):
            sim = cosine_similarity(valid_grads[ids[i]], valid_grads[ids[j]])
            if sim > config.cosine_threshold:
                duplicate_groups.append((ids[i], ids[j]))

    for group in duplicate_groups:
        # 保留质押额更高的节点，标记其余为复制
        sorted_nodes = sorted(group, key=lambda x: stake[x])
        for node_id in sorted_nodes[1:]:
            results[node_id] = "REJECTED: suspected_gradient_copy"
            del valid_grads[node_id]

    # 3. 对抗方向检测
    if len(valid_grads) >= 3:
        aggregate = mean(valid_grads.values())
        for node_id, grad in valid_grads.items():
            cos = cosine_similarity(grad, aggregate)
            if cos < config.adversarial_threshold:
                results[node_id] = "REJECTED: suspected_adversarial"
                del valid_grads[node_id]

    return results, valid_grads
```

### 3.4 贡献评分验证（模型提升）

最终判断训练是否有效，取决于该节点的梯度对全局模型性能的边际贡献。

详见第四节"奖励分配机制"。

### 3.5 验证时间线

一轮训练的完整验证时间线：

```

│  训练节点 │  提交梯度+    │  聚合节点   │  贡献评分  │  奖励
│  本地训练 │  ZK证明       │  统计检测   │  计算     │  分发
│           │               │  + 聚合     │           │
│           │               │             │           │
│  ~15min   │  合约验证     │  ~30min     │  ~10min   │  链上

## 四、奖励分配机制

### 4.1 奖励来源与结构

```
任务奖池 (Task Reward Pool)
├── 训练奖励 (Training Reward) ────── 70%  ── 按贡献分给训练节点
├── 聚合奖励 (Aggregation Reward) ── 15%  ── 分给聚合节点
├── 数据奖励 (Data Reward) ───────── 10%  ── 按数据使用量分给数据提供者
└── 挑战奖励 (Challenge Reserve) ─── 5%   ── 奖励有效挑战者
```

### 4.2 训练节点奖励：基于近似Shapley值

**为什么选Shapley值**：Shapley值是合作博弈论中唯一满足效率性、对称性、虚拟玩家性和可加性的贡献分配方法，是公平分配的理论最优解。

**精确Shapley值的问题**：计算精确Shapley值需要枚举所有2^n个联盟子集，对n个训练节点来说是指数复杂度，不可行。

**近似方案：边际贡献采样法（MC-Shapley）**

核心思想：通过随机排列采样来近似Shapley值。

```
算法：MC-Shapley 近似

输入：训练节点集合 N = {1, 2, ..., n}
      全局模型评估函数 V(S) = model_performance(aggregate(gradients_S))
      采样次数 M

对每个节点 i:
    φ_i = 0

对 m = 1 到 M:
    随机生成排列 π = (π_1, π_2, ..., π_n)
    S = ∅
    对 j = 1 到 n:
        S_prev = S
        S = S ∪ {π_j}
        marginal = V(S) - V(S_prev)
        φ_{π_j} += marginal

对每个节点 i:
    φ_i = φ_i / M

输出：{φ_1, φ_2, ..., φ_n}
```

**MVP简化方案——Leave-One-Out（LOO）近似**：

在MVP阶段，采用更简单的LOO近似：

```
φ_i ≈ V(N) - V(N \ {i})

即：节点i的贡献 = 全部节点参与时的模型性能 - 排除节点i后的模型性能
```

LOO是Shapley值的一阶近似，复杂度O(n)，对于小规模节点（≤10）足够精确。

**贡献评分的具体执行**：

1. 聚合节点在链下执行：
   - 计算全量聚合模型性能 `V(N)`
   - 对每个节点i，计算排除该节点后的聚合模型性能 `V(N\{i})`
   - 计算LOO贡献 `φ_i = V(N) - V(N\{i})`
2. 聚合节点将贡献评分结果和计算过程的ZK证明提交至合约
3. 合约验证证明后，按贡献比例分配该轮训练奖励

**负贡献处理**：

若某节点的LOO贡献为负（即加入该节点反而降低了模型性能），该节点：
- 不获得训练奖励
- 扣除10%质押金
- 信誉分-10
- 该节点的梯度不参与最终聚合

### 4.3 聚合节点奖励

聚合节点的奖励由两部分组成：

| 组成 | 计算方式 | 说明 |
|:---|:---|:---|
| 基础奖励 | 任务奖池的15% | 完成聚合即获得 |
| 绩效奖金 | 额外奖池的0~5% | 如果聚合后模型性能超过任务发起者设定的目标阈值 |

聚合节点需在规定时间内提交聚合结果，超时则降为基础奖励的50%。

### 4.4 数据提供者奖励

数据奖励按数据使用量和数据质量分配：

```
data_reward_i = data_pool × (usage_i × quality_i) / Σ(usage_j × quality_j)
```

- `usage_i`：节点i的数据被训练节点使用的次数
- `quality_i`：数据质量评分（初始为1.0，根据使用该数据的训练结果动态调整）

**数据质量自校正机制**：

如果使用某数据提供者数据的训练节点持续产出负贡献梯度，该数据的质量评分自动下降：

```
quality_i = quality_i × decay_factor    # decay_factor = 0.9 per negative round
quality_i = max(quality_i, 0.1)         # 下限保护
```

### 4.5 奖励发放时间线

```
轮次结束
    │
    ├─ 即时：统计检测通过的节点获得基础质押释放
    │
    ├─ T+1轮：贡献评分计算完成，训练奖励按比例释放
    │
    ├─ T+7天：聚合节点挑战期结束，质押+聚合奖励释放
    │
    └─ 任务结束时：数据提供者按累计使用量获得数据奖励
```

### 4.6 特殊情况处理

| 情况 | 处理方式 |
|:---|:---|
| 所有节点贡献为负 | 本轮无人获得训练奖励，奖金回流至任务奖池 |
| 仅1个节点贡献为正 | 该节点获得全部训练奖励，其余节点按负贡献处理 |
| 聚合节点作恶导致无法计算贡献 | 由挑战者重新聚合并计算贡献，原聚合节点质押罚没 |
| 任务中途终止 | 已完成的轮次按正常流程分配，未开始轮次的奖金退还任务发起者 |

### 5.3 关键合约交互流程

#### 5.3.1 一轮训练的完整合约调用序列

```
1. task.startRound()
   ├── 检查 task.status == Active
   ├── currentRound++
   ├── 设置 roundStartTime = block.timestamp
   ├── 设置 submissionDeadline = block.timestamp + SUBMISSION_WINDOW
   └── emit RoundStarted(currentRound)

2. trainer.submitGradient(gradientHash, zkProof, dataCommitment, modelSnapshotHash)
   ├── 检查 msg.sender 已质押且信誉 ≥ 10
   ├── 检查 block.timestamp < submissionDeadline
   ├── 验证 zkProof（调用验证器合约）
   ├── 验证 modelSnapshotHash == 当前全局模型哈希
   ├── 存储 TrainerSubmission
   └── emit GradientSubmitted(trainer, currentRound)

3. aggregator.submitAggregation(newModelHash, performanceHash, aggregationProof, contributionScores)
   ├── 检查 msg.sender 是当前轮次选举的聚合节点
   ├── 检查 block.timestamp > submissionDeadline（提交窗口已关闭）
   ├── 检查 block.timestamp < aggregationDeadline
   ├── 验证 aggregationProof
   ├── 存储 AggregationResult
   └── emit AggregationSubmitted(currentRound)

4. task.endRound()
   ├── 检查聚合结果已提交或超时
   ├── 对每个训练节点：
   │   ├── 如果质量检测通过 → 标记为有效
   │   └── 如果质量检测失败 → 触发 slash()
   ├── 根据贡献评分分配奖励
   ├── 更新全局模型哈希
   ├── 铸造模型版本NFT
   └── emit RoundCompleted(currentRound)

5. trainer.claimReward(round)
   ├── 检查该轮次该节点的贡献评分 > 0
   ├── 计算应得奖励 = roundReward × (φ_i / Σφ)
   ├── 转账奖励
   └── emit RewardClaimed(trainer, amount)
```

#### 5.3.2 挑战流程的合约调用

```
1. challenger.challengeAggregation(round, challengeStake)
   ├── 检查 challengeStake >= minChallengeStake
   ├── 冻结聚合节点质押
   ├── 设置 challengeDeadline = block.timestamp + CHALLENGE_WINDOW
   ├── task.status = Challenged
   └── emit AggregationChallenged(round, challenger)

2. challenger/agg submitArbitration(round, independentAggregationProof)
   ├── 双方各提交独立聚合证明
   ├── 合约比较两份证明
   └── 如果挑战者正确 → slash(aggregator, 100%), 奖励挑战者
       如果被挑战者正确 → slash(challenger, challengeStake), 归还被挑战者

3. task.resolveDispute(round)
   ├── 恢复 task.status = Active
   ├── 使用正确的聚合结果继续
   └── emit DisputeResolved(round, winner)
```

### 5.4 合约安全考虑

| 安全维度 | 措施 |
|:---|:---|
| 重入攻击 | 所有外部调用使用 Checks-Effects-Interactions 模式，关键函数加 ReentrancyGuard |
| 整数溢出 | 使用 Solidity 0.8+ 内置溢出检查 |
| 权限控制 | 角色权限分离（ADMIN, AGGREGATOR, TRAINER），使用 AccessControl |
| 前运行攻击 | 提交阶段使用 commit-reveal 模式，先提交哈希后揭示梯度 |
| 拒绝服务 | 限制单轮最大参与者数量，设置超时自动推进机制 |
| ZK验证Gas优化 | 使用递归证明压缩，链上仅需验证一个常量大小的证明 |
| 升级安全 | 关键合约使用 UUPS 代理模式，升级需多签+时间锁 |

## 七、聚合节点选举与轮换

### 7.1 选举机制

每轮训练开始时，通过可验证随机函数（VRF）从候选聚合节点中选举一名。

```
选举算法：

1. 候选聚合节点集合 A = {a_1, a_2, ..., a_k}
2. 每个候选的权重 w_i = stake_i × reputation_i
3. 使用 Chainlink VRF 生成随机数 r
4. 计算累积权重 C_j = Σ_{i=1}^{j} w_i
5. 选中节点 = a_m，其中 m 是满足 C_m ≥ r × C_k 的最小 m
```

### 7.2 轮换规则

- 每轮选举一个新的聚合节点
- 同一聚合节点不能连续当选超过3轮
- 聚合节点需在规定时间内完成聚合（默认2小时），否则视为放弃
- 放弃的聚合节点扣除30%质押，重新选举

### 7.3 聚合节点职责

1. 收集本轮所有验证通过的梯度
2. 执行统计质量检测
3. 执行联邦平均聚合
4. 计算LOO贡献评分
5. 提交聚合结果和贡献评分证明
6. 铸造新的模型版本NFT

## 八、隐私保护设计

### 8.1 数据隐私

| 层级 | 保护方式 | 安全保证 |
|:---|:---|:---|
| 传输层 | TLS + 端到端加密 | 网络传输中数据不可被窃听 |
| 存储层 | AES-256-GCM 加密存储 | 数据存储不可被未授权方读取 |
| 使用层 | TEE（Intel SGX/AMD SEV） | 训练时数据仅在可信执行环境中解密 |
| 验证层 | ZK证明 | 验证方无需看到原始数据即可验证计算正确性 |

### 8.2 模型隐私

- 全局模型参数存储在IPFS，仅通过哈希在链上引用
- 模型访问权限由任务发起者控制（可设为私有/公开/付费）
- 梯度更新可选择使用同态加密传输给聚合节点（MVP暂不实现，留作后续）

### 8.3 梯度隐私（防梯度推断攻击）

**问题**：研究表明，从梯度更新中可以推断出训练数据的部分信息（梯度推断攻击）。

**防护方案**：

- **差分隐私（DP）**：训练节点在提交梯度前添加 calibrated Gaussian 噪声
- **梯度裁剪**：将梯度L2范数裁剪到固定上限C，限制单样本对梯度的最大影响
- **安全聚合（MVP后）**：使用Bonawitz等人提出的安全聚合协议，聚合节点只能看到聚合后的梯度，无法看到单个节点的原始梯度

```
差分隐私梯度提交流程：

原始梯度 g
    │
    ├─ 裁剪：g' = g × min(1, C / ||g||)
    │
    ├─ 加噪：g'' = g' + N(0, σ²C²I)   (σ由目标ε, δ决定)
    │
    └─ 提交：hash(g'') + ZK证明(g'')
```

                      │    └────────────┘
                      │          │
                      │          │ 争议解决后继续
                      │          ▼
                      │    ┌────────────┐
                      └────│   Active   │
                           └─────┬──────┘
                                 │ 达到目标或最大轮次
                                 ▼
                           ┌────────────┐
                           │ Completed  │
                           └────────────┘

另：Created → Cancelled（发起者主动取消）
```

### 9.2 任务参数

| 参数 | 说明 | 默认值 |
|:---|:---|:---|
| `maxRounds` | 最大训练轮次 | 100 |
| `roundDuration` | 每轮持续时间（含训练+提交+聚合） | 4小时 |
| `submissionWindow` | 梯度提交窗口 | 2小时 |
| `aggregationWindow` | 聚合窗口 | 1小时 |
| `challengeWindow` | 挑战窗口 | 7天 |
| `minTrainers` | 最少训练节点数 | 3 |
| `maxTrainers` | 最大训练节点数 | 50 |
| `convergenceThreshold` | 收敛阈值（连续N轮性能提升<δ则终止） | δ=0.1%, N=5 |
| `performanceTarget` | 性能目标（如准确率>95%） | 由任务发起者设定 |

### 9.3 任务终止条件

| 条件 | 触发方式 |
|:---|:---|
| 达到最大轮次 | 自动 |
| 达到性能目标 | 自动 |
| 收敛判定（连续5轮性能提升<0.1%） | 自动 |
| 任务发起者主动终止 | 手动 |
| 奖励池耗尽 | 自动 |

### 10.2 模型血缘链

```
Model v1.0 (初始)
    │
    ├── Model v1.1 (Round 1: Alice, Bob)
    │       │
    │       ├── Model v1.2 (Round 2: Alice, Bob, Charlie)
    │       │       │
    │       │       └── Model v1.3 (Round 3: Bob, Charlie, Dave)
    │       │
    │       └── Model v1.2' (Round 2 fork: Alice, Dave)  ← 挑战导致分叉
    │
    └── Model v1.1' (Round 1 fork: Charlie, Dave)  ← 不同数据分区
```

NFT的 `parentVersion` 字段构成模型血缘的有向无环图（DAG），实现完全可追溯。

## 十二、技术选型与实现细节

### 12.1 技术栈

| 层级 | 技术选型 | 理由 |
|:---|:---|:---|
| 区块链 | Ethereum L2 (Base/Arbitrum) | 低Gas费、高TPS、以太坊安全性 |
| 智能合约 | Solidity 0.8.x + Foundry | 成熟生态、丰富工具链 |
| ZK证明 | RISC Zero zkVM (Rust) | 通用ZK虚拟机，可直接验证Python/PyTorch执行 |
| 训练框架 | PyTorch | 最广泛使用的ML框架 |
| 链下计算 | Python + RISC Zero Guest | 训练逻辑在zkVM guest中执行 |
| 存储 | IPFS + Filecoin | 去中心化存储模型和数据 |
| 前端 | Next.js + wagmi + viem | 现代Web3开发栈 |
| 聚合服务 | Rust | 高性能梯度聚合和贡献计算 |
| 随机数 | Chainlink VRF v2 | 可验证随机性 |

### 12.2 RISC Zero zkVM集成方案

```
┌─────────────────────────────────────────────┐
│  Host 程序 (Rust)                           │
│  ├── 加载模型参数                           │
│  ├── 加载训练数据                           │
│  ├── 调用 zkVM guest 执行训练               │
│  └── 获取证明和journal                      │
├─────────────────────────────────────────────┤
│  Guest 程序 (Rust)                          │
│  ├── 接收模型参数 + 数据                    │
│  ├── 前向传播 → 计算损失                    │
│  ├── 反向传播 → 计算梯度                    │
│  ├── 哈希关键值作为公开输出                 │
│  │   ├── model_hash                         │
│  │   ├── data_commitment                    │
│  │   ├── gradient_hash                      │
│  │   └── hyperparams_hash                   │
│  └── 输出 journal (公开输出)                │
├─────────────────────────────────────────────┤
│  证明验证 (Solidity Verifier)               │
│  ├── 输入：proof + journal                  │
│  ├── 调用 RISC Zero 链上验证器              │

```

## 十三、风险评估与缓解

| 风险 | 等级 | 缓解措施 |
|:---|:---|:---|
| ZK证明生成耗时过长影响用户体验 | 高 | MVP使用小型模型；后续优化电路、支持递归证明 |
| 聚合节点中心化风险 | 中 | VRF随机选举+质押加权+轮换限制 |
| 梯度推断攻击导致数据泄露 | 中 | 差分隐私+梯度裁剪+未来支持安全聚合 |
| 女巫攻击（多账号） | 中 | 质押门槛+余弦相似度检测+地址行为分析 |
| Gas费过高 | 中 | 部署在L2；递归证明压缩；批量操作 |
| 模型收敛缓慢 | 低 | 动态调整学习率；支持异步聚合 |
| 数据提供者造假 | 低 | 数据质量自校正机制；信誉惩罚 |
